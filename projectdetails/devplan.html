Development Plan

Relational Schema

We will use the following relations.

  • Boxers(netid, name, experience, handedness, school_year, picture_url, weight, dorm, email)
    • netid → name, experience, handedness, school_year, picture_url, weight, dorm, email
    • Key: netid
    • Superkeys: any combination that includes netid
  • Spar_Signups(date, ring, spar_number, boxer1_netid, boxer2_netid)
    • date, ring, spar_number → boxer1_netid, boxer2_netid
    • Key: date, ring, spar_number
    • Superkeys: any combination that includes the three elements of the key
  • Spar_Results(num_spar_for_boxer, boxer_netid, date, comments, opponent, score, coach)
    • num_spar_for_boxer, boxer_netid → date, comments, opponent, score, coach
    • Key: num_spar_for_boxer, boxer_netid
    • Superkeys: any combination that includes the two elements of the key
  • Accounts(netid, password_hash, permissions)
    • Passwords will be stored in an encrypted/hashed form
    • netid → password_hash, permissions
    • Key: netid

Databases/Platforms/Languages

We will be using the mysql database system that is on the dsg1 server. We will use HTML, Node, and React for the web programming portion. For any intermediate programming/ data processing/ algorithm-writing, we plan on using Python.

Getting Data

We will get our data from Notre Dame’s boxing club. We have talked to the boxing head coach and have access to spar sheet data from the last ~5 years; we may end up only inserting one year of spar data into our database, depending on time constraints. One year of data is also approximately 150 spar sheets (one per boxer) with an average of 10 spar entries per sheet. This means one year of data has around 1000 separate spar entries, which will most likely be plenty of data. In order to enter the data into the database, we will have to manually enter the paper data into a digital format. We plan on hand-entering data into a csv file, and then transferring that data programmatically into our database.

To store data in our database, we plan on using fake names to go with real spar data. We will still have a password system in order to be able to see the full list of boxers, but our entered data will have names obscured for privacy/security reasons. If this system goes live we would get users’ consent to entering their data, but for the way this is set up we’re not going to take any chances and just protect the data with obscurity.

Labor Division

Entering the data into a csv/the database is a task that will be split evenly between the members of the group.

The current plan is to do a lot of together-programming, where we all work in the same location on similar tasks and divide things up into smaller individual tasks after working as a group. We all want to become at least slightly proficient at all the aspects of the project, from the web programming to the database interactions and processing. We plan on dividing up the database interactions into one person taking entering, querying, and deleting data.

We also plan on dividing up the main webpages and advanced functions of the project like so:

  • Query page, bracket analysis functions (Madalyn)
  • Spar sign up page, spar alternate/analytics functions (Alison)
  • Bracket page, email function (Grace)
